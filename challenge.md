# Challenge A: Building a Mini Cloudgeni DSL to IaC Translator

**Duration**: 2 hours

**Objective**: The goal is to test the candidate's ability to design, implement, and reason about a domain-specific language (DSL) and its transformation into Infrastructure-as-Code (IaC) for a basic cloud infrastructure setup.

**Challenge Description**

You are tasked with creating a simplified version of the Cloudgeni DSL that can:

- Define basic infrastructure components.
- Translate the DSL into a working IaC (Terraform, Pulumi, or OpenTofu) template.

### Task Requirements
**Define a Simple DSL Schema:**

Create a sample schema for a DSL that describes infrastructure components, including:
Database: Specify type (e.g., MySQL), instance size, and region.
VM: Specify instance type, operating system, and region.
Storage: Specify size, type (block or object), and region.
Network: Specify subnet, CIDR block, and region.
Example DSL input (JSON or YAML):

```bash
yaml

database:
  type: mysql
  size: small
  region: us-east-1
vm:
  instance_type: t2.micro
  os: ubuntu
  region: us-east-1
storage:
  size: 50GB
  type: block
  region: us-east-1
network:
  cidr: 10.0.0.0/16
  region: us-east-1
```

**API to Generate DSL:**

Create a simple RESTful API endpoint (e.g., /generate-dsl) that accepts a brief prompt describing infrastructure requirements and outputs the DSL schema in JSON or YAML.
Use any LLM provider or API behind the scenes as necessery.

Example input to the API:

```bash
"Create a MySQL database, Ubuntu VM, 50GB block storage, and a private network."
Example output from the API:
(as defined in the DSL schema above)
```

**Basic DSL to IaC Renderer:**

Write a script or program to convert the DSL schema into IaC in one of the following formats:
Terraform
Pulumi
OpenTofu
Example Terraform output (partial):
```bash
hcl
resource "aws_instance" "vm" {
  ami           = "ami-0abcdef1234567890"
  instance_type = "t2.micro"
}
```

**Infrastructure Target:**

The IaC should target Azure, AWS, or GCP (choose one for simplicity).

### Expectations

**Code Quality:** The solution should be structured, modular, and include clear separation of concerns.

**Completeness:**

- DSL schema is well-defined.
- API works and generates DSL based on prompts.
- Basic rendering of DSL to IaC is functional.
  
**Execution:**

While the infrastructure doesn't need to be deployed, the IaC output should be syntactically correct for the chosen cloud provider.

**Deliverables:**

- A GitHub PR containing the solution.
- Include clear instructions in a README.md file for how to run the API and renderer.
  
**Example Outputs:**
- Sample DSL schema generated by the API.
- Corresponding IaC template generated from the renderer.
  
**Evaluation Criteria**

Correctness: Does the output match the input specifications?

Creativity: How well is the problem approached within the time limit?

Usability: Is the API intuitive? Are the IaC templates valid?

Efficiency: Is the code well-optimized and organized?

This challenge tests your ability to build scalable and modular components while solving a real-world cloud infrastructure problem. Good luck!




# Challenge B: Enhancing LLM Accuracy for IaC Generation Using Agentic Workflows
**Duration:** 2 hours

**Objective:** The goal is to test the candidate's ability to design, implement, and optimize workflows for an LLM to produce more accurate Infrastructure-as-Code (IaC) output directly from user prompts. The challenge involves using techniques like few-shot prompting, pseudo-retrieval-augmented generation (pseudo-RAG), and retry mechanisms.

**Challenge Description:**
You are tasked with creating a system that improves LLM output accuracy for generating IaC directly from a natural language prompt. This involves building a simple pipeline where the LLM:

- Uses few-shot prompting to generate IaC output.
- Enhances the response using a pseudo-RAG mechanism (e.g., reference examples or pre-defined templates).
- Incorporates a retry mechanism to address incorrect or incomplete IaC output.

### Task Requirements
**1. Few-Shot Prompting System:**

Set up a series of examples (at least 2–3) to guide the LLM’s understanding of infrastructure prompts and their expected IaC output.
Use these examples to improve IaC generation accuracy.
Example Inputs and Outputs for Few-Shot Prompting:

Example 1:
```
Input Prompt:

Create an Ubuntu VM with 4 CPUs and 16GB RAM on AWS in the us-west-2 region.
```
Expected Output (Terraform):
```
hcl
resource "aws_instance" "vm" {
  ami           = "ami-0abcdef1234567890"
  instance_type = "t2.xlarge"
  availability_zone = "us-west-2a"
}
```

Example 2:
```
Input Prompt:

Deploy an S3 bucket with versioning enabled in the ap-southeast-1 region.

```
Expected Output (Terraform):

```
hcl
resource "aws_s3_bucket" "example" {
  bucket = "example-bucket"
  versioning {
    enabled = true
  }
  region = "ap-southeast-1"
}
```
**2. Pseudo-RAG Mechanism:**

Implement a way to augment LLM generation by referring to a repository of pre-defined IaC snippets or schema examples.
The system should use these snippets to validate or enhance the generated IaC output.

**3. Retry Mechanism:**

Implement logic to detect invalid or incomplete IaC output (e.g., missing fields, syntax errors).
Allow the system to automatically re-prompt the LLM with a revised query or provide additional context to improve the output.

Example Retry Flow:
```
Initial LLM output:
hcl
resource "aws_instance" "vm" {
  instance_type = "t2.micro"
}
```
Retry mechanism detects a missing ami field.
Revised prompt another agent to LLM:

```
Include the correct AMI for Ubuntu in the us-west-2 region.
```
Corrected output:
```
hcl
resource "aws_instance" "vm" {
  ami           = "ami-0abcdef1234567890"
  instance_type = "t2.micro"
}

```
**4. Infrastructure Target:**

Choose one cloud provider (Azure, AWS, or GCP) and ensure the IaC is valid for deployment.


### Expectations
**Code Quality:**

The solution should be modular and well-commented.
**Completeness:**

The few-shot prompting system improves initial accuracy.
Pseudo-RAG enhances the LLM output quality using reference examples.
The retry mechanism corrects errors and ensures completeness.

**Execution:**

The IaC output must be syntactically correct for the chosen cloud provider.

**Deliverables:**
A GitHub PR containing the solution.
Include clear instructions in a README.md file for how to run the system and test its functionality.

Example Outputs:

Input prompts, LLM outputs (initial and final), and reference examples used.


**Evaluation Criteria**

Correctness: Does the final IaC output match the input specifications?

Efficiency: How well does the system handle retries and optimize LLM interactions?

Creativity: How well are techniques like pseudo-RAG and few-shot prompting implemented?

Usability: Is the pipeline easy to use and adapt for future improvements?

This challenge tests your ability to build intelligent workflows that enhance LLM capabilities for real-world IaC generation. Good luck!
